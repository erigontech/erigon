# Snapshots Management

Erigon stores the majority of its chain data in **snapshot files** (`.seg` segments and associated index files). These immutable, compressed files cover headers, bodies, transactions, and state history. Because snapshots can occupy hundreds of gigabytes, understanding what is on disk — and how much space each category consumes — is essential for capacity planning and node maintenance.

## `seg du` — Disk Usage Report

The `seg du` subcommand scans the snapshot directory for a given `--datadir` and reports a breakdown of disk usage by category, the detected (and configured) node type, and estimated sizes for each prune mode.

### Syntax

```bash
erigon seg du --datadir <path> [--json] [-v | --verbose]
```

### Flags

| Flag | Description |
|------|-------------|
| `--datadir <path>` | Path to the Erigon data directory. Defaults to the standard per-network data directory when omitted. |
| `--json` | Output results as JSON instead of human-readable text. Useful for scripting or dashboards. |
| `-v`, `--verbose` | Show a per-domain / per-type subcategory breakdown within each category. |

### Example output (human-readable)

```
mainnet | full | blocks 0–22.000.000 | steps 0–18.500
total: 1.2TB (42810 files)

── Breakdown ──────────────────────────────────────────────────
  domains                 612.0GB      48.8%  18200 files
  block segments          410.4GB      32.7%  12400 files
  history                 120.7GB       9.6%   6300 files
  inverted indices         88.3GB       7.0%   2100 files
  accessors                21.0GB       1.7%    910 files
  caplin                    1.2GB       0.1%     90 files
  other                     0.5GB       0.0%      8 files

── Estimated Size by Node Type ────────────────────────────────
  archive        1.3TB          —    all blocks       all history
  full           1.2TB     -70.0GB    last 262.144     last 262.144
  blocks         1.3TB     -30.0GB    all blocks       last 262.144
  minimal        1.1TB    -150.0GB    last 100.000     last 100.000
```

Sizes use Erigon's `ByteCount` format (e.g. `850.4GB`, `1.2TB` — binary scale, no space). Block/step counts and prune-window sizes use `.` as the thousands separator, matching the binary's own output.

**Header fields explained:**

- **Chain** — detected from `chaindata` (falls back to `unknown` if the database is locked or empty).
- **Mode** — the configured prune mode read from the database, plus the mode detected from which files are present on disk. If they disagree, both are shown (e.g., `full (files look like archive)`).
- **blocks 0–N** — the highest block number across all block segment files.
- **steps 0–N** — the highest step number across all state files (steps are an internal Erigon unit; one step ≈ a fixed number of blocks, depending on chain activity).

**Breakdown categories:**

The category labels below are exactly what the command prints.

| Category | Contents |
|----------|----------|
| `domains` | Current-state domain files (latest accounts, storage, code, commitment). Never pruned, so usually the largest category. |
| `block segments` | Headers, bodies, and transactions (`.seg` + index files) directly under `snapshots/`. |
| `history` | State-history snapshots for accounts, storage, code. |
| `inverted indices` | Inverted indices over state history (enables range lookups). |
| `accessors` | Accessor index files that speed up state queries. |
| `caplin` | Consensus-layer (beacon chain) snapshots. |
| `commitment hist` | Commitment-history files (archive nodes only). |
| `rcache` | Receipt cache (pre-computed receipts for RPC performance). |
| `forkable` | Forkable (unwindable) snapshot data. |
| `other` | Any files with unrecognised extensions (listed inline). |

**Estimates table:** shows how much space the current snapshot set would occupy under each prune mode, given the files already on disk. The `delta` column is relative to archive (negative = smaller than archive). This lets you compare your current footprint against what other modes would consume without resyncing.

### Example output (verbose, `-v`)

With `--verbose`, each category expands into its domain or type subcategories:

```
  history                 120.7GB       9.6%   6300 files
  ├─ accounts          60.3GB      50.0%   3150 files
  ├─ storage           48.3GB      40.0%   2520 files
  └─ code              12.1GB      10.0%    630 files
```

### JSON output (`--json`)

Use `--json` to get a machine-readable object suitable for scripting:

```bash
erigon seg du --datadir /data/erigon --json | jq '.estimates'
```

The JSON schema matches the `duResult` struct: top-level fields `chain`, `configured_mode`, `detected_mode`, `block_range`, `step_range`, `total_bytes`, `total_files`, `categories`, `subcategories`, `other_extensions`, `estimates`.

:::tip
Run `seg du` while the node is **stopped**. If Erigon is running and holds the database lock, the command falls back to `unknown` for the chain name and omits the configured mode — it will still scan snapshot files and produce size figures, but mode information will be incomplete.
:::

## Snapshot Types (Prune Modes)

The snapshot set on disk reflects the node's **prune mode**. `seg du` automatically detects which mode your current files correspond to. For full details on choosing a prune mode, see [Prune Modes](prune-modes).

| Mode | History retained | Transactions retained | Typical use case |
|------|------------------|-----------------------|------------------|
| **Archive** | All history from genesis | All blocks | Developers, RPC providers, researchers needing full historical state. |
| **Full** (default) | Last 262,144 blocks (~36 days, EIP-8252 window) | Last 262,144 blocks | General users, DApp interaction, fastest sync. |
| **Blocks** | Last 262,144 blocks | All blocks (genesis onward) | Users needing full block/receipt history without full historical state. |
| **Minimal** | Last 100,000 blocks (~14 days) | Last 100,000 blocks | Solo staking, hardware-constrained setups, maximum privacy. |

The `seg du` estimates table shows the projected disk footprint for each mode given the files currently present, so you can evaluate a mode change before committing to a resync.

## Upgrading Snapshots in v3.5

### EIP-8252 retention window change

Erigon v3.5 aligns the **Full** and **Blocks** prune modes with [EIP-8252](https://github.com/ethereum/EIPs/pull/11601), which defines `REORG_RETENTION_WINDOW = 2^18 = 262,144 blocks` (≈ 36.4 days) as the inactivity-leak-bounded non-finality window. Prior versions used a retention window of **100,000 blocks** (~14 days).

**What this means for existing nodes:**

- Nodes previously running in **Full** or **Blocks** mode now retain a larger history window (262,144 blocks instead of 100,000).
- On first startup after upgrading to v3.5, Erigon **automatically migrates** your snapshot configuration — no manual intervention is required for most setups.
- Disk usage for Full and Blocks nodes will increase slightly over time as the larger retention window fills in. Use `seg du` to monitor the change.

**Minimal mode is unchanged** — it continues to use a 100,000-block window and is not subject to EIP-8252 compliance requirements.

### Manual snapshot operations

To migrate snapshots to the latest format, reset them for maximum performance after a major upgrade, or downgrade to an older snapshot format, follow the step-by-step snapshot upgrade and downgrade instructions in the [Upgrading guide](../get-started/installation/upgrading). Those commands are subcommands of `erigon snapshots` (e.g. `erigon snapshots update-to-new-ver-format --datadir /your/datadir`).

:::warning
Always **back up your `--datadir`** before running snapshot reset, upgrade, or downgrade commands.
:::
